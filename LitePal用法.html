一.LitePal介绍

LitePal是郭霖开源的Android数据库框架,它采用了对象关系映射(ORM)的模式,并将我们平时开发最常用到的一些数据库功能进行了封装,使得不用编写SQL语句就可以完成各种建表和增删改查的操作，并且LitePal很“轻”，jar包大小不到100k，而且近乎零配置，这一点和Hibernate这类的框架有很大区别。

作者的关于此的CSDN博客

源码

框架使用体验

  1. 简单易上手,CURD操作封装详尽
  2. 框架仍在更新,今年添加了对索引和事务的支持
  3. xml文件的创建需要符合规范
  4. .db文件放在外部存储时需要特别注意
  5. 更新表结构时需要更新版本号或者删除应用重新安装
  6. 不支持代码修改数据库和表结构

  • 实体类说明

    1. 自增长主键id的类型只能为int或者long,如无ID字段LitePal会自动添加

    2. ID下标从1开始自增长无法修改

    3. 只有private修饰的字段才会被映射到数据库表中

    4. LitePal支持的数据类型一共有8种，int、short、long、float、double、boolean、String和Date

    5. Column注解

      1. 唯一 : unique = true
      2. 非空: nullable = true
      3. 默认值 : defaultValue= "默认值"; 只能用以String类型的字段
      4. 忽略: ignore = true; 忽略即不在数据库中创建该属性对应的字段

十.索引

  • 郭霖关于是否使用索引的分析

    xxxxxxxxxx
    5
    1
    我们到底应不应该使用索引？
    2
    ​
    3
    其实应不应该使用要看你到底用不用得着，我的个人看法是绝大部分人应该是用不着的，因为移动端的数据库几乎不太可能会存储这么多的数据。而在用不着的情况下强行使用索引，反而可能会降低你的其他数据库操作的效率（比如增删改），因为维护索引的B+树也是需要耗时的。
    4
    ​
    5
    另外，添加索引的列尽量要保证数据重复率非常低才行，不然索引将会失去效果。这个也很好理解，比如我给性别这一列加上索引，由于性别就只有男女两种，1000万条数据我可能要查询500万条才能把指定性别的所有数据查询出来，这种索引基本是没有作用的。
